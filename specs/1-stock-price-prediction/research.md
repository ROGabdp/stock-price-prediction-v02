### 2. 機器學習函式庫選型

- **待研究**: 評估並選擇適合股價預測的機器學習函式庫。
- **決策**: TensorFlow/PyTorch
- **理由**: 深度學習框架，適合處理複雜的時間序列數據，並能支援更進階的模型架構以提升預測準確性。
- **考慮的替代方案**: scikit-learn (通用、易用), Prophet (時間序列預測)

### 3. Web 框架選型

- **待研究**: 評估並選擇適合全棧 Python Web 應用程式的 Web 框架。
- **決策**: Flask
- **理由**: 輕量級且彈性高，適合快速開發和客製化，能與各種前端技術和機器學習後端良好整合。
- **考慮的替代方案**: Django (功能齊全、ORM), FastAPI (高效能、API 優先)

### 4. UI 框架選型

- **待研究**: 評估並選擇適合 Python Web 應用程式的 UI 框架，以實現互動式介面和圖表展示。
- **決策**: Dash
- **理由**: Dash 基於 Plotly，專為互動式儀表板和資料視覺化設計，能與 Flask 後端良好整合，非常適合展示股價數據和預測結果。
- **考慮的替代方案**: Streamlit (快速開發、資料科學導向), 傳統前端技術 (例如 HTML/CSS/JavaScript 搭配 Flask)

### 5. 繪圖函式庫選型

- **待研究**: 評估並選擇適合 Web 介面中展示歷史數據和預測結果的繪圖函式庫。
- **決策**: Plotly
- **理由**: Plotly 提供高度互動式的圖表，與 Dash 框架原生整合，非常適合在 Web 應用中展示複雜的股價數據和預測趨勢。
- **考慮的替代方案**: Matplotlib (基礎、靈活), Seaborn (基於 Matplotlib、美觀)

### 6. 模型元資料儲存方案

- **待研究**: 確定儲存已訓練模型元資料（例如：模型名稱、訓練日期、使用的資料集、效能指標）的最佳方式。
- **決策**: JSON 檔案
- **理由**: 簡單、易讀，適合儲存非複雜的結構化資料，且與 Python 的資料結構轉換方便，符合輕量級專案的需求。
- **考慮的替代方案**: SQLite (輕量級資料庫), 專用模型管理工具 (例如 MLflow)
